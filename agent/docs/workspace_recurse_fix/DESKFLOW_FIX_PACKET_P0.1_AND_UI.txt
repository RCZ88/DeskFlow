---------------------------------------------------------------------
WHY P0 DID NOT WORK (corrected root cause, grounded in opencode docs+issues)
---------------------------------------------------------------------
opencode's TUI is a Bubble Tea full-screen app. In its input box:
  * Enter  = CR (0x0D) = SUBMIT the message.
  * Newline insert = Ctrl+J (0x0A) / Shift+Enter. (opencode docs: "Keybinds";
    Shift+Enter is unreliable across terminals, Ctrl+J is the safe newline.)
  * Bracketed paste (\x1b[200~ ... \x1b[201~) is UNRELIABLE in opencode v1
    (known regression; Ctrl+V/paste partly unimplemented). opencode GitHub
    issue #23524: when a multi-line blob arrives, "each \n triggers a separate
    submit event -> multiple messages sent instead of one."

That is EXACTLY the "pushes down like Enter in a Word doc" symptom:
our init prompt contains newlines (\r\n / \n). Each newline is read by
opencode as a SUBMIT, so it fires many submits, re-rendering and scrolling
the conversation downward, and never types the prompt as one block.

P0 wrapped the text in bracketed paste + a trailing \r. Because opencode v1
does not honor bracketed paste reliably, the embedded newlines STILL submit
per line. So P0 could not fix this. We must change the ENCODING, not wrap it.

Second cause (push-down even before any prompt, at launch): we launch
`opencode -s <resumeId>` with a resume id on a brand-new session, and/or we
write to the PTY during the launch window before the TUI has taken over.
That garbles the initial render. CZ's instinct is correct: launch opencode
FRESH, let it create the session, and only resume with -s when we actually
have a known existing session id.

SECTION A — FIX TERMINAL INPUT ENCODING  (THE BLOCKER)
File: src/main.ts  (helper buildAgentInputPayload added in P0)
=====================================================================
WHY: Newlines in the payload submit per-line in opencode. Bracketed paste
is unreliable. We must send interior line breaks as Ctrl+J (0x0A) and submit
exactly once with a single trailing CR (0x0D).

CHANGE — rewrite buildAgentInputPayload(text, agent) so that, for agents
whose AGENT_CONFIGS entry is a TUI (opencode AND claude):
  1. Normalize first:  text = text.replace(/\r\n?/g, "\n")   // kill all CR
  2. Trim trailing newlines/spaces from the end.
  3. DO NOT wrap in bracketed paste. Remove the \x1b[200~ / \x1b[201~ wrap.
  4. Body = the normalized text where every interior "\n" stays as "\n"
     (0x0A = Ctrl+J = newline-insert in opencode). Do NOT convert "\n" to
     "\r". There must be ZERO carriage returns inside the body.
  5. Submit: append exactly ONE "\r" (0x0D) at the very end.
     Final payload = body + "\r".
  For NON-TUI / plain shell agents (if any), keep prior behavior (data + "\r").

IMPORTANT: also fix agent:send so it does NOT append its own "\r\n".
The single trailing \r must come only from buildAgentInputPayload. Search for
any remaining `+ '\r\n'` or `+ "\n"` on the agent write path and remove it.

OPTIONAL ROBUSTNESS (hands may evaluate): chunk the body so opencode's
textarea ingests it as keystrokes, e.g. write body, then after a ~40ms
settle write the lone \r. Only add if a single write still mis-submits.

DONE WHEN:
  - Sending a 3-paragraph prompt results in ONE message in opencode, typed
    as a single multi-line block, submitted once. No per-line scroll/push.
  - No \x1b[200~ / \x1b[201~ and no \r\n remain on the agent write path.

SECTION B — LAUNCH FRESH, GATE READY, NO PREMATURE RESUME / WRITES
Files: src/main.ts (launch + ready), src/pages/TerminalPage.tsx (initializeTerminal)
=====================================================================
WHY: Launching `opencode -s <resumeId>` on a new session with a stale/empty
id, plus writing during the launch window, garbles the TUI (push-down at
launch, before any prompt).

CHANGE:
  B1. New-session launch command: launch the agent FRESH with no -s flag.
      i.e. build `cd <projectPath> && opencode`  (NOT `opencode -s <id>`).
      Only append `-s <resumeId>` when ALL are true: user explicitly chose
      "resume", resumeId is a non-empty validated id, and that session is
      known to exist. Otherwise never pass -s.
  B2. Do NOT write ANYTHING to the PTY between spawn and ready. Queue the
      init prompt in pendingWrites and flush ONLY after markAgentReady().
  B3. After ready fires, wait a short settle (~250-400ms) before the first
      flush so the TUI has fully grabbed the PTY, then flushPendingAgentWrites().
  B4. Resume id capture is OUT OF SCOPE here. Do not synthesize a resume id.
      (Note for later: opencode sessions live in
      ~/.local/share/opencode/opencode.db; resume retrieval is a future packet.)

DONE WHEN:
  - Creating a session runs `cd <dir> && opencode` with NO -s on first launch.
  - opencode renders its prompt cleanly with no push-down at launch.
  - The init prompt is sent only once, after ready + settle.
